Before we get to our CSS, a brief history and Where We Are Now

The Old Way: 3rd Party Modules

Image of old file structure, lots of separated CSS files 3rd party modules have a lot of their own opinions about css and markup. getting these to play nice with our own theme is not always possible; some modules aren't templatable (can't change the markup) or not themable (can't change the css). You're left with a lot of overriding and !important hell

The Old Way: Custom Modules

Our custom modules sometimes fell into the same pitfall: Scatter module.css files with unshareable styles.

The New Way: 3rd Party Modules

Choose Better 3rd Party Modules Look for ones that are templatable and themable are the templates and theme files easilly maintained in version control? (i.e. can you only edit them in some weird DNN interface) Look for ones the follow modern approaches to web design: Do they work in a responsive setting? Do they use semantic HTML? (i.E. are there tables for the layout?)

The New Way: 3rd Party Modules (cont)

Some of our favorite modules EasyDNNNews News/Events/Content Types Ventrian News Articles News/Content Types 2SexyContent Everything ever!

The New Way: Custom Modules and 3rd Party Modules

No more module.css files. No more scattered style sheets specific to a module. All of our styles are maintained through our theme.

Now that that's cleared up, let's talk about our CSS

OOCSS

- "Separate Structure and Skin" - Define certain features (such as a combination of padding and border-color) as small css "objects" that can be built up and combined with other objects to form modules. Also, avoid styling elements directly. Bad / Good - "Separate Container and Content" - CSS objects should be movable; don't rely on an objects location to style it. Bad / Good

A Little Dash of SMACSS

We don't follow it to a T, but here are our take-a-ways: - Minimize Depth of Applicability - Ours is shallow, we use a lot of classes on our elements and create a lot of modules because it makes it that much easier to scale. - Separate our CSS into Base Rules, layout rules and Modules (and even then, some of our layouts ARE more like modules)

Sass

We use Sass because it lets us write scalable OOCSS more efficiently and it makes it easier to maintain. - mixins are a lifesaver - placeholders are awesome - @extend is an OOCSS-er's best Friend
